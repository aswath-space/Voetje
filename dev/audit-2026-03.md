## 🔴 Critical — Will Cause Bugs in Production

### C-1 · `assert` stripped in release builds — EnergyCalculator.dart:43

```dart
assert(price > 0, 'Price per kWh must be positive for $countryCode');
return cost / price;
```

`assert` statements are **completely removed** by Flutter's AOT compiler in release builds. If `EnergyDefaults.getPricePerKwh()` ever returns `0` for an unsupported or future country code, the division by zero happens silently. The result is `Infinity`, which flows into the CO2 calculation, into the database, and onto the dashboard — all without any error.

**Fix:** Replace with a runtime check and a fallback default price, or throw a proper exception.

---

### C-2 · `importFromJson` has no error handling and no deduplication — database_service.dart:310

Two separate problems in the same method:

**No try/catch on `jsonDecode`:** A malformed or truncated backup file will throw an unhandled exception. Since this is called from a UI action, it crashes the app with no user-facing message.

**No deduplication:** If the user taps "Import" twice with the same file, all entries are doubled in the database. There is no idempotency key, no duplicate check, no "already imported" detection. For a user who has 200 entries and imports twice, they now show 400 entries with identical data. There is no way to undo this short of "Clear all data."

**Fix:** Wrap in try/catch and return an error string. For deduplication, compare `(date, category, subCategory, value)` against existing entries before inserting, or add an `exported_at` fingerprint to the JSON and refuse to re-import the same export.

---

### C-3 · Static `_database` field shared across all `DatabaseService` instances — database_service.dart:13

```dart
static Database? _database;
```

```dart
DatabaseService.forTesting(Database db) {
  _database = db;  // mutates the CLASS-LEVEL static
}
```

`_database` is a static field, so all `DatabaseService` instances share the same reference. The `forTesting` constructor replaces the class-level database. In practice this works because there's only one `DatabaseService` in the app, but it's a landmine: if any test forgets to reset state, or if a second `DatabaseService()` is ever constructed (say, in a future feature), both instances silently use whichever database was initialised last. The testing constructor is also setting a real static that persists across tests.

**Fix:** Convert `_database` to an instance variable. Pass a `Database?` through the normal constructor for testing, or use a factory that accepts an injection.

## 🟠 High — Calculation and Logic Flaws

### H-1 · Train emission factor inconsistency between code and documentation

- **Code** (`transport_mode.dart:41`): `kgCO2PerKm: 0.041`
- **Documentation** (`dev/emission-factors/transport.md`): `0.035 kg CO2/km`

These don't match. One of them is wrong, or the code was updated without updating the docs. DEFRA 2023 national rail is 0.035 kg/km. The description also says "National rail average" which is UK-specific language while the app targets international users.

**Impact:** Every train entry since launch has been logged at 0.041 instead of 0.035 — an overestimate of ~17%.

### H-2 · `_todayCO2` headline number is internally inconsistent

In `refreshData()` (emission_provider.dart):
- **Transport**: actual today's logged CO2 ✓
- **Food**: actual today's logged CO2 ✓
- **Energy**: monthly average ÷ 30 (daily avg) — **included in `_todayCO2`**
- **Shopping**: monthly total ÷ 30 — **NOT included in `_todayCO2`**, only shown in breakdown row
- **Waste**: weekly total — **NOT included in `_todayCO2`**, only shown in breakdown row

So the headline "Today's Footprint" number includes an energy daily average but silently excludes shopping and waste averages. The dashboard breakdown row shows all categories but with different time periods (today / daily avg / monthly avg / weekly), which is correct. The problem is the progress bar toward the Paris budget uses this inconsistent `_todayCO2`, so a user who is only tracking waste and shopping will see a progress bar that doesn't reflect their full picture. This is documented in code comments but is a methodological flaw that should be either fixed (include all categories' daily averages) or explicitly surfaced to the user ("Based on transport + food + energy").

### H-3 · `EnergyCalculator.quickEstimate` fallback divides by 30 — emission_provider.dart:294

```dart
_energyDailyAvgCO2 = EnergyCalculator.quickEstimate(...) / 30;
```

`EnergyCalculator.dailyAverage()` (the correct version) divides by the actual number of days in the month. The fallback divides by a hardcoded 30. February has 28/29 days, meaning the daily average shown in February is ~7% too low. July/August/etc. have 31 days, meaning the daily average is ~3% too high.

### H-4 · GridIntensity silently falls back to US average for unsupported countries

```dart
return byCountry[code] ?? byCountry['US']!;  // 390 g/kWh
```

The grid intensity table has 20 countries. For everyone else — Nigeria, Turkey, Egypt, Saudi Arabia, most of Africa, Southeast Asia, all of Latin America except Brazil — the fallback is the US average (390 g/kWh). This is wrong in both directions: Nigeria's grid is ~450, Turkey ~380, Egypt ~380, but several Scandinavian users not in the table get 390 when they should get 20-50.

There is no warning to the user that their country isn't supported and a fallback is being used.

**Impact:** Systematically wrong energy CO2 for users outside the 20 supported countries, with no indication that this is happening.

### H-5 · `getLifetimeSecondHandSavings` duplicated between provider and database

`DatabaseService.getShoppingLifetimeSavings()` (line 289) computes the savings in SQL — single query, efficient. `EmissionProvider.getLifetimeSecondHandSavings()` (line 521) fetches ALL shopping entries into Dart memory and computes the same calculation in a fold. The provider version loads unbounded rows. If a user has thousands of shopping entries (unlikely but possible), this is a memory spike for no reason. The SQL version already exists and is correct.

The provider method is never called from anywhere in the UI (the dashboard uses `_lifetimeSecondHandSavings` which comes from the SQL path). The Dart version appears to be dead code.

## 🟡 Medium — Performance and Maintainability

### M-1 · `Consumer<EmissionProvider>` wraps entire dashboard — home_screen.dart:167

`_DashboardContent` uses `Consumer<EmissionProvider>` which rebuilds **the entire scroll view** — all stat cards, the chart, all category rows, all recent entries — whenever any single field in `EmissionProvider` calls `notifyListeners()`. This happens on every `addEntry()`, `deleteEntry()`, `toggleHabit()`, `refreshData()`, `setPreferredUnit()`, etc.

The `_DashboardTab` wrapper above it correctly uses `Selector` to guard the initial loading/error state, but once it delegates to `_DashboardContent`, granularity is lost. Each sub-widget reading from the provider should use `Selector` or `context.select()` to narrow its rebuild scope.

**Performance impact:** On the home screen, every habit toggle triggers a full dashboard rebuild including the chart widget.

---

### M-2 · Airport search is O(n) linear scan on every keystroke — airport_service.dart:65

Every character typed in the airport picker iterates over the full `_cache` (3,500 airports) three times (IATA scan, city scan, name scan). That's ~10,500 comparisons per keystroke. On a mid-range Android device this is fine, but it could be improved to ~O(1) for IATA lookups (already done with `_iataIndex`) and ~O(log n) for prefix searches with a sorted structure.

More practically: there's no debounce on the text field (it needs to be checked in `airport_picker.dart`). If the user types quickly, each intermediate character triggers a new search.

### M-3 · `_isCarMode` and passenger-picker logic duplicated across two files

`TransportMode._isCarMode` getter (transport_mode.dart:148-155) and `_selectMode` in add_entry_screen.dart (lines 296-303) both contain an identical hardcoded list of car modes. If `carElectric` or any car variant is ever added or renamed, both lists need updating. They will silently diverge. The `_isCarMode` getter exists for exactly this purpose — `_selectMode` should use it instead of duplicating the list.

## ✅ What's Done Well

These are genuinely good engineering decisions that deserve recognition:

- **`Future.wait` for concurrent DB queries** — `refreshData()` launches all queries in parallel via a named-futures map. This is the right approach and significantly faster than sequential awaits.
- **`batchReplace` transaction** — delete + insert in a single SQLite transaction. No partial states.
- **`context.mounted` guards** — consistently checked before using `context` after every `await`. No dangling context references found.
- **Foreign key cascade** — `route_presets.from_place_id` and `to_place_id` reference `saved_places(id) ON DELETE CASCADE`, and `PRAGMA foreign_keys = ON` is set on every DB open. Deleting a saved place cleans up its route presets automatically.
- **`AirportService` lazy loading** — airports.json is only loaded when flight mode is first tapped, not on form open. IATA strings are persisted as strings in SharedPreferences and resolved to `Airport` objects only after the JSON is in memory.
- **`Selector` on the dashboard loading/error state** — `_DashboardTab` uses `Selector` to avoid rebuilding the full dashboard for unrelated state changes. Partially undone by the full `Consumer` inside `_DashboardContent`, but the intent is right.
- **`HabitLog` streak logic** — the "today or yesterday" grace period for streaks is correct UX. Users don't lose their streak because they haven't logged the habit at midnight.
- **Database schema versioning** — `migrateSchema` handles all version transitions without dropping tables. Users upgrading from v1 won't lose data.
- **`copyWith` on `EmissionEntry`** — immutable model with `copyWith`. Correct Dart pattern.
- **`getShoppingLifetimeSavings` SQL aggregation** — uses `SUM(CASE WHEN ...)` in SQL rather than loading all rows. Correct approach.

### CD-1 · 🔴 Critical — Electric car uses a hardcoded UK grid factor, not local grid intensity

**File:** `transport_mode.dart:73`

```dart
carElectric(
  label: 'Electric Car',
  kgCO2PerKm: 0.047,
  description: 'BEV average (grid electricity)',
),
```

`0.047 kg CO2/km` is the DEFRA 2023 UK-grid BEV figure (the actual DEFRA 2023 value is `0.053` — the `0.047` appears to be the 2024/2025 updated value as the UK grid decarbonises). Regardless of which DEFRA year it represents, **this factor is UK-specific and is applied globally.**

The app already has `GridIntensity.forCountry()` and uses it correctly in `EnergyCalculator.electricityCO2()`. The electric car should do the same calculation: `avgEVEfficiencyKWhPerKm × gridIntensity / 1000`.

**Impact by country:**

| Country | Hardcoded factor | Correct factor (at ~200 Wh/km) | Error |
|---------|-----------------|-------------------------------|-------|
| UK | 0.047 | 210 × 0.2 / 1000 = 0.042 | ~+12% |
| Norway | 0.047 | 20 × 0.2 / 1000 = 0.004 | **+1,075%** |
| France | 0.047 | 60 × 0.2 / 1000 = 0.012 | **+292%** |
| Poland | 0.047 | 750 × 0.2 / 1000 = 0.150 | **−69% (underestimate)** |
| South Africa | 0.047 | 900 × 0.2 / 1000 = 0.180 | **−74% (underestimate)** |

A Norwegian EV driver is shown emitting 10× more than they actually do. A South African EV driver is shown as 4× cleaner than reality. This is the single most egregious factor error in the codebase.

**Note on EV efficiency assumption:** Standard BEV efficiency ranges from 150–250 Wh/km depending on vehicle size. A midpoint of 200 Wh/km (0.2 kWh/km) is defensible as a global average. The factor should be `GridIntensity.forCountry(userCountryCode) * 0.2 / 1000`.

The difficulty: transport mode doesn't currently accept a country code. The country code lives in `EnergyProfile`, not directly accessible from the transport entry screen. The fix requires either passing country code into `calculateCO2()` for EV modes, or computing the EV factor at the time of logging using the profile's country code.

### CD-2 · 🟠 High — Household energy scale factors assume `annualKwh` is a single-person baseline, but it's a household total

**Files:** `energy_defaults.dart`, `energy_calculator.dart`

```dart
// annualKwh stores household TOTALS (from IEA household consumption data):
'GB': 3800,  // UK average HOUSEHOLD = 3,800 kWh (for ~2.4 people average)
'US': 10500, // US average HOUSEHOLD = 10,500 kWh

// householdScale is then applied per person:
static const Map<int, double> _householdScales = {
  1: 1.0, 2: 0.65, 3: 0.50,
};
// 5+ people: 0.35 default
```

The scale logic reads as: "a 1-person household uses 100% of the national average household's energy." But the national average household has ~2.4 people. This assigns the entire household footprint to a single person.

**Concrete UK numbers:**

| Household size | Code computes | Actual UK average | Error |
|---------------|--------------|-------------------|-------|
| 1 person | 3,800 kWh → 798 kg CO2/yr | ~1,800 kWh → 378 kg CO2/yr | **+111%** |
| 2 people | each 2,470 kWh → 519 kg CO2/yr | each ~1,600 kWh → 336 kg CO2/yr | **+54%** |
| 3 people | each 1,900 kWh → 399 kg CO2/yr | each ~1,250 kWh → 263 kg CO2/yr | **+52%** |
| 4 people | each 1,330 kWh → 280 kg CO2/yr | each ~1,050 kWh → 221 kg CO2/yr | **+27%** |

The overestimate is largest for small households — the most common user profile (solo urban dwellers, couples). The root fix is either:

**Option A:** Keep `annualKwh` as household total but use correct per-person scales based on household size relative to national average. E.g. UK 1-person household uses ~47% of the household average (1,800 / 3,800).

**Option B:** Replace `annualKwh` with per-capita values (`annualKwhPerCapita`) and remove `householdScale` entirely, since per-capita values already represent one person.

### CD-4 · 🟠 High — Shopping items mix lifecycle and manufacture-only CO2 without any consistency

**File:** `item_catalog.dart`

The `co2KgNew` values appear to pull from multiple methodologies without standardisation:

| Item | Code value | Apparent source | Method |
|------|-----------|-----------------|--------|
| Jeans | 33 kg | Levi's 2015 LCA | **Full lifecycle** (incl. use-phase washing/drying) |
| T-shirt | 7 kg | Fashion industry avg | **Full lifecycle** (incl. use-phase) |
| Smartphone | 70 kg | Apple/Samsung env. reports | Manufacture + transport only |
| Laptop | 350 kg | Dell/Lenovo LCA (range) | Manufacture + transport only |
| Laptop (Apple MacBook M-series) | ~147 kg (Apple report) | Manufacture only | Manufacture only |

The Levi's jeans figure of 33 kg is specifically their own lifecycle analysis including ~37 washes over the garment's life. Manufacture-only for a pair of jeans is approximately 12–15 kg CO2e. Using 33 kg inflates jeans CO2 by ~2.2× relative to a manufacture-only methodology.

If the app is tracking **purchase emissions** (which is the appropriate scope for a shopping tracker), all items should use manufacture + supply chain CO2 only, not include use-phase washing/maintenance which the user will generate regardless of purchase.

The 350 kg laptop figure also appears to be at the high end. Apple MacBook M-series laptops (popular category) report 147–203 kg CO2e. A generic "laptop" at 350 kg may be reasonable for large gaming laptops but over-represents the average by ~50–75%.

**Specific correction needed:**
- Jeans: 33 kg → ~14 kg (manufacture only, consistent methodology)
- T-shirt: 7 kg → ~3 kg (manufacture only; the 7 kg figure is lifecycle)
- Laptop: 350 kg is defensible for a large Windows laptop but overstates MacBook/ultrabook category

### CD-5 · 🟠 High — `ShoppingCondition.repaired` (10%) attributed more CO2 than `secondHand` (5%)

**File:** `shopping_item.dart:1-9`

```dart
enum ShoppingCondition {
  newItem('New', 1.0),
  secondHand('Second-hand', 0.05),
  repaired('Repaired', 0.10);
```

The multipliers attribute higher emissions to **repair** than to **second-hand purchase**. This implies:
- Second-hand laptop: 350 × 0.05 = **17.5 kg CO2**
- Repaired laptop: 350 × 0.10 = **35 kg CO2**

Environmentally, repair is generally considered the more sustainable option — it avoids a transaction entirely, uses fewer new materials, and extends product life. Showing repair as twice as dirty as buying second-hand contradicts core messaging for an environmental tracking app.

The 5% for second-hand is intended to model delivery/transport emissions of a second-hand purchase. The 10% for repair is intended to model parts + labour materials. However, the repair multiplier is applied to the **new item's full manufacturing CO2** as a base (e.g. 10% of 350 kg = 35 kg), which is methodologically strange — repair materials don't scale with the manufacturing footprint of the original item.

Both multipliers are fairly arbitrary. Either swap them (repaired < secondHand) or use absolute fixed values for transport/repair emissions rather than a percentage of new-item CO2.

### CD-6 · 🟡 Medium — `_weeklyWasteCO2` can be negative; UI handling unclear

**File:** `waste_setup.dart:5`, `emission_provider.dart:316`

```dart
BinType.recycling('Recycling', '♻️', -0.30),  // negative CO2 per kg
...
_weeklyWasteCO2 = wasteEntries.fold(0.0, (s, e) => s + e.co2Kg);
```

Recycling entries have negative `co2Kg` values (−0.30 per kg of material diverted). A user who recycles 10 kg of materials generates `10 × −0.30 = −3.0 kg CO2` from recycling. If they also put out 3 kg of general waste (`3 × 0.58 = +1.74 kg CO2`), `_weeklyWasteCO2 = −3.0 + 1.74 = −1.26 kg`.

This represents a genuine net saving and the negative value is *conceptually correct*. The question is whether the UI handles it. If `WasteDashboardCard` displays the raw `weeklyWasteCO2` value as a number or in a chart without a sign/colour change to distinguish savings from emissions, users will see a negative number on a "CO2 emitted" card, which is confusing.

Additionally: `_weekCO2` (the total weekly CO2 SQL query, line 226) **also** sums all entries including negative waste entries. This means heavy recycling genuinely reduces the weekly total, which may surprise users who expect `weekCO2` to only count positive emissions.

### CD-8 · 🟡 Medium — Smartphone charge equivalency uses an incorrect constant

**File:** `emission_calculator.dart:17`

```dart
'smartphones': '${(co2Kg / 0.008).toStringAsFixed(0)} smartphone charges',
```

Using `0.008 kg CO2` as the CO2 per smartphone charge (i.e. one full charge ≈ 0.008 kg CO2e).

A smartphone charges from 0–100% using approximately 8–12 Wh (let's call it 10 Wh = 0.01 kWh). At the US average grid (390 g/kWh): `0.01 kWh × 0.390 = 0.0039 kg CO2`. At the UK grid (210 g/kWh): `0.01 × 0.210 = 0.0021 kg CO2`.

Using 0.008 kg/charge would require a grid of 800 g/kWh — higher than Poland (750) or South Africa (900, but only at those extremes). For the overwhelming majority of users, 0.008 is ~2–4× too high.

**Effect:** The comparison reads as fewer charges than it should. 1 kg CO2 = `1/0.008 = 125 smartphone charges`. At the correct ~0.004 kg/charge value: 1 kg CO2 = 250 smartphone charges. The equivalency understates the scale of impact by half.

The US EPA's own smartphone charge calculator quotes ~0.005 kg CO2 per full charge (US grid). The 0.008 constant appears to originate from an older or non-standard source.

### CD-9 · 🟡 Medium — `MealSlot.dinner` window starts at 14:00, encompassing all afternoon meals

**File:** `meal_type.dart:31-34`

```dart
if (h >= 5 && h < 10) return MealSlot.breakfast;
if (h >= 10 && h < 14) return MealSlot.lunch;
if (h >= 14 && h < 21) return MealSlot.dinner;  // 2pm onwards is "dinner"
return MealSlot.snack;
```

A meal logged at 14:00 (2pm), 15:00 (3pm), 16:00 (4pm), or 17:00 (5pm) is automatically labelled as "Dinner 🌙." Most people eating at these times are having a late lunch, afternoon snack, or early dinner. The standard dinner window across most cultures is 17:00–21:00.

This does not affect CO2 calculation (slot label has no bearing on emission factor), but it affects the `TodaysMealsCard` grouping and meal slot distribution display. A user's "lunch" logged at 1:55pm appears as "Lunch" but the same meal at 2:05pm appears as "Dinner."

**Suggested boundary:** lunch 10:00–15:00, dinner 15:00–21:00 (or 17:00–21:00 with an afternoon "snack" slot for 14:00–17:00).

## Data Layer Consolidation — 2026-03-18

### Problem

Reference data (country profiles, emission factors, physical constants) was scattered across 14 files with no single source of truth. Specific issues:

1. **Triple-maintained country list** — `energy_defaults.dart` (23 entries), `energy_setup_screen.dart` (21 entries), and `grid_intensity.dart` (24 entries) had independent, diverging country lists. Nigeria and Pakistan had data but were invisible in the setup picker.
2. **Country data split across 5 maps in 2 files** — adding one country required editing `gridIntensity`, `annualKwh`, `annualGasKwh`, `electricityPricePerKwh`, `gasPricePerKwh`, and `countryNames` across `grid_intensity.dart` and `energy_defaults.dart`. Missing any one map caused silent fallback errors.
3. **Scattered constants** — gas combustion factor in `energy_calculator.dart`, waste density in `waste_calculator.dart`, smartphone charge in `emission_calculator.dart`, EV efficiency in `transport_mode.dart`, Paris budget in `emission_calculator.dart`, household scales in `energy_calculator.dart`.

### P3-H1 · 🟠 High — `HeatingType.oil` and `HeatingType.wood` are collected in setup but silently contribute zero CO2 to any calculation

**Files:** `energy_profile.dart:4–9`, `energy_calculator.dart`, `emission_factors.dart`

The `HeatingType` enum exposes four options to users in the energy setup screen:

```dart
enum HeatingType {
  electric('Electric heater/AC', '⚡'),
  gas('Gas boiler', '🔥'),
  oil('Oil heating', '🛢️'),
  wood('Wood/pellets', '🪵'),
  notSure('Not sure', '🤷');
```

`EnergyCalculator.quickEstimate` — the only path that computes a CO2 estimate from profile data — models electricity and gas only:

```dart
double householdCO2 = electricityCO2(kWh: monthlyElecKwh, countryCode: countryCode);
if (country.hasGas) {
  householdCO2 += gasCO2(kWh: monthlyGasKwh);
}
```

There is no code path for oil or wood combustion anywhere in the codebase. `EmissionFactors` has no `oilKgCO2PerKwh` or `woodKgCO2PerKwh` constant. A user who selects "Oil heating 🛢️" receives an estimate computed as if their home has zero heating emissions from that source.

**Real-world impact by fuel:**

| Fuel | DEFRA 2023 factor | Typical household use | Annual error |
|------|-----------------|-----------------------|-------------|
| Heating oil | 0.298 kg CO2/kWh | ~18,000 kWh/year (UK) | **~5,364 kg CO2 omitted** |
| Wood/pellets | 0.016 kg CO2/kWh (biomass, net of LULUCF) | ~15,000 kWh/year | ~240 kg CO2 omitted |

Oil heating homes are common in rural UK, Ireland (~10% of households), Germany, and much of North America off the gas grid. The omission causes the biggest undercount of any remaining issue in the codebase — an oil-heated household's energy estimate may show near-zero when the reality is 5+ tonnes CO2/year from heating alone.

**Note on wood:** The UK DEFRA biogenic accounting treats wood as near-carbon-neutral at the combustion step (residual factor 0.016 kg/kWh net), but the user should still see a value rather than zero. Some methodologies count wood combustion at gross ~0.39 kg CO2/kWh and apply a separate credit.

**Fix required:** Add `oilKgCO2PerKwh` and `woodKgCO2PerKwh` constants to `EmissionFactors`. In `quickEstimate` (and in bill-entry mode), check `heatingTypes` from the profile and sum contributions from oil/wood using country-average consumption figures. Alternatively, surface a data entry screen for oil/wood volume (analogous to the electricity/gas bill entry).

### P3-L2 · 🔵 Low — `isCarMode` allocates a new `List` on every call

**File:** `transport_mode.dart:160–167`

```dart
bool get isCarMode => [
  TransportMode.carSmall,
  TransportMode.carMedium,
  TransportMode.carLarge,
  TransportMode.carElectric,
  TransportMode.carHybrid,
  TransportMode.taxi,
].contains(this);
```

The list literal `[carSmall, carMedium, ...]` creates a new `List<TransportMode>` object on each invocation. `isCarMode` is called inside `calculateCO2()`, which is called on every keystroke in the distance field. The Dart VM will GC these quickly, but the allocation is unnecessary.

- Declare a `static const Set<TransportMode> _carModes = {...}` and use `_carModes.contains(this)`.
- Or use a switch expression: `switch (this) { carSmall || carMedium || ... => true, _ => false }`.
